clang_llvm

发表于 2021-05-20 更新于 2021-05-21 分类于 clang ， llvm

背景

​ 对于刚开始接触clang编译器的时候，对于clang、llvm这个两个工具是分不清楚的。根据官方资料，刚开始llvm是用来实现虚拟机的工具，所以llvm相当于有自己的机器码，这就是IR。llvm包含了一系列的工具与库，方便开发者用来构建编译器和just-in-time编译器。llvm只能编译符合自己标准的IR语言，所以如果需要编译编程语言的化，需要工具将编程语言转化为IR，这个工具被称作为编译器前端。

​ clang是一个完整的c/c++/object-c编译器，它包含了clang 前端（将c/c++/object-c源码转换为llvm IR）,llvm 后端。同时，clang还是一个编译器驱动，当给clang工具指定不同参数的时候，它会内部调用llvm的工具，直接生成最终的目标。

llvm 设计思想

llvm的设计思想有如下几个方面：

  • llvm最经典的设计就是使用分层设计思想来支持不同的编程语言的编译
  • 有用自己的IR
  • 使用Pass

现在通过如上所讲的三个特点来分析llvm的优势。

分层设计思想

llvm设计为前端、优化器、后端。其中后面两个部分是包含在llvm代码中的。由于后面两个部分基本固定，所以对于开发新的编程语言的编译器来说，只要设计出前端就可以了。下面结合下面的图来说明：

不同语言的编译器，可以共用优化器与后端。下面分别介绍这个三个层次的作用。

  1. 前端

前端是用于生成llvm IR的一个工具，例如clang，能够将c/c++/object-c语言转换成llvm IR，其中包括检查程序文件语法、检测语法错误、生成AST、最终生成llvm IR。例如llgo用于将go语言转换成llvm IR。

  1. llvm优化器

llvm优化器用于优化LLVM IR。经过优化后的llvm IR执行性能更高。

  1. 后端

llvm 优化器的输出还是llvm IR，所以如果需要编译成对应的目标平台机器代码需要后端的参与。后端有两种情况，生成对应的目标平台的机器码(.o)或者汇编码。所以如果选择生成汇编码，用户还可以使用gcc之类的工具来进行编译。(ps: 即使最终生成的是目标平台二进制文件，但是如果需要经过链接，那么还是需要借助binutils中的ld工具进行连接，不过现在llvm自带的llvm-ld工具也比较好用了)。

llvm IR

llvm IR是llvm唯一能够识别的语言(还记得前面介绍的llvm 最初设计为虚拟机吧，跟jvm一样，llvm也有自己的字节码bitcode)。因为llvm框架最终处理的都是llvm IR，所以如果需要使用llvm框架，只需要实现对应语言的前端，将对应语言转换为llvm IR即可，大大减少了编译器开发的难度。llvm可以运行llvm IR，前提是需要将llvm IR转换为bitcode，这个可以通过llvm-as工具实现，编译生成的bitcode就可以在llvm中运行了。由于llvm IR具有简单，可读性强等特点，所以也减轻了开发前端与优化器的难度。

llvm Pass

llvm pass也体现了llvm的设计思想，llvm将不同的功能设计成独立的模块，可以单独编译成库文件，方便在使用的时候进行加载，这样的模块就叫做Pass，它可以是一个优化算法，也可以是一个埋点功能等等。多个Pass之间应该是不耦合的，如果一个Pass的输出是另一个Pass的输入，这种有依赖的Pass在加载的时候，会将两个Pass都加载进来。Pass的引入也是为了开发的简单，更进一步降低了使用llvm框架开发的难度。

memory-test

发表于 2021-05-12 更新于 2021-05-13 分类于 Linux

关于Linux中对内存耗用测试时，VSS/RSS/PSS/USS 的介绍

VSS - Virtual Set Size （用处不大）

虚拟耗用内存（包含共享库占用的全部内存，以及分配但未使用的内存）。其大小还包括了可能不在物理RAM中的内存（比如malloc分配了内存空间，但是还未被使用）。VSS很少被用于判断一个进程的真实内存使用量。

RSS - Resident Set Size （用处不大）

实际使用物理内存（包含共享库占用的全部内存）。但是RSS还是可能会造成误导和干扰，因为他仅仅表示该进程所使用的所有共享库的大小，它不管有多少个进程使用该共享库，该共享库仅被加载到内存一次。所以RSS并不能准确反映单进程的内存占用情况。

PSS - Proportional Set Size （用处不大）

实际使用的物理内存（比例分配共享库占用的内存，按照进程数等比例划分）。例如，如果有三个进程都使用了同一个共享库，共占用了30页的内存。那么PSS将认为每个进程分别占用该共享库10页的大小。PSS 是非常有用的数据，因为系统中的所有的进程的PSS都相加的话，就刚好反映了系统中的总共占用的内存。而当一个进程被销毁之后，其占用的共享库那部分比例的PSS，将会再次按比例分配给余下使用该库的进程。这样PSS可能会造成一些误导，因为当一个进程被销毁之后，PSS不能准确的表示返回给全局系统的内存。

USS - Unique Set Size (非常有用)

进程独自占用的物理内存（不包含共享库占用的内存）。USS是非常有用的数据，因为他反映了运行一个特定进程真实的边际成本（增量成本）。当一个进程被销毁后，USS是真是返回给系统的内存。当进程中存在一个可疑的内存泄露时，USS是最佳观察数据。

Android的procrank工具可以用来查看VSS、RSS、PSS、USS。

zip 文件详解

发表于 2021-04-14 分类于 zip ， tools
介绍

.zip文件是存储多个文件的归档文件。ZIP允许使用多种压缩算法压缩包含的文件，同时也支持简单地存储文件而不进行压缩(只进行归档)。.zip中的每个文件是单独存储的，所以是允许使用不同是压缩算法对同一zip包中的不同文件进行压缩(支持不同文件使用不同的压缩算法)。由于ZIP归档文件中的文件是单独压缩的，因此可以提取文件或添加新文件，而无需对整个归档文件进行压缩或解压缩。这与压缩成的tar格式文件形成对比，压缩后的tar文件不容易进行这种随机访问处理。
在zip文件的末尾，存放了一个目录结构。这个目录结构标识了zip中包含的文件，并标识了这些文件在zip归档文件中的位置。这样，ZIP阅读器就可以加载文件列表，而无需读取整个ZIP归档文件。ZIP归档文件还可以包含与ZIP归档文件无关的额外数据。通过将程序代码添加到ZIP归档文件，并将文件标记为可执行文件，可以将ZIP归档文件制作成自解压归档文件（用于解压缩其包含的数据的应用程序）。将目录存储在最后还可以通过将压缩文件附加到无害文件（例如GIF图像文件）中来隐藏压缩文件。
.ZIP格式使用32位CRC算法，并包括归档目录结构的两个副本，以提供更大的保护，防止数据丢失。

文件结构

zip文件很容易标识，因为在zip文件的结尾是一个central directory record的结构，因为这个结构在文件结尾,所以往zip中添加新文件很方便。如果在zip文件结尾的central directory record结构标识该文件是一个非空归档文件，那么该zip中被压缩的每个文件、目录都应该存在一个central directory entry结构，这个结构包含了该文件或目录的其它元数据（文件名、目录名
、实际被压缩文件偏移量等）。有了这个central directory entry，就可以轻松的定位到被压缩的文件，因为不必阅读整个档案即可查看文件列表。central directory entry中还包含了local file header的信息。因为zip中可以附加文件，所以只有在central directory record中有记录的文件才是有效文件。扫描整个zip文件中的local file header来确认文件数是无效的（除非归档文件已损坏），因为central directory record可能会声明某些文件已被删除而其他文件已被更新。

c++运算符重载

发表于 2021-03-03 分类于 c++

c++的运算符是通过函数来实现的(每个运算符都有一个运算符函数)。通过在运算符上使用函数重载，可以自定义运算符的实现，以用于不同的类型（比如自定义的类）。这就是运算符重载。

编译器内部附带了一个+号运算符函数来处理整数的求和运算(这个函数对x和y求和，然后返回求和的结果)。类似于编译器在看到x+y表达式的时候，就会调用operator+(int,int)运算符函数。

#####C++如何实现运算符重载
当编译器在计算已给包含运算符的表达式的值的时候，编译器使用如下规则处理：

  • 如果所有的操作数都是基本数据类型，编译器将调用内部附带的运算符函数（如果有的话）。如果这个运算符函数不存在，那么编译器将产生一个编译错误。
  • 如果表达式中有一个或多个操作数是程序员自定义的类型(例如自定义的类、枚举等)，编译器将查看该类型是否具有可调用的匹配重载运算符函数。如果找不到，它将尝试将一个或多个用户定义的类型操作数转换为基本数据类型，以便它可以使用匹配的内置运算符函数。如果失败，则将产生编译错误。

#####C++运算符重载的限制

  • 基本所有的c++运算符都可以重载。除了：条件运算符(?:),sizeof,作用域运算符(::),成员选择运算符(.),成员指针选择运算符(.*),typeid运算符,还有casting运算符(类型转换运算符)。
  • 只允许重载c++语法中存在的运算符。不能够创建新的运算符、或者重命名已经存在的运算符(例如不能创建一个新的@运算符来实现新功能)。
  • 运算符重载中，至少需要有一个操作符类型是自定义的。
  • 在运算符重载中，操作数的个数不能超过运算符所支持的个数。
  • 所有的运算符在重载后，优先级跟c++原生支持的优先级保持一致，不能更改。

考虑一个情况：如果想通过重载异或运算符^实现幂运算，例如4+3^2，按照意图结果应该是4+9=13,但是由于在c++中，+的优先级比^优先级高，所以结果是(4+3)^2=49。当然这里这个例子有点问题，对基本数据类型是不能重载运算符的，不过可以将基本数据类型封装在自定义类中。

#####使用c++运算符重载应该遵循的规则

  1. 在重载运算符的时候，重载的功能最好能够跟运算符本来的意义相似。比如重载+算符符，那么重载后的能够应该就是对两个对象求和。
  2. 如果对运算符重载的功能意图不是很明确，也就是运算符不能够清晰的表明重载的意图，应该是用成员函数代替运算符重载。

c++ friend 关键字

发表于 2021-03-01 更新于 2021-03-02 分类于 c++

c++中friend关键字用于使得不在本类中的成员函数能够访问本类的私有成员变量。分为三种情况:

  • 某个外部函数，利用friend关键字能够访问本类中的私有成员变量
  • 另一个类中的所有成员函数通过friend关键字能够访问本类中的私有成员变量。
  • 另一个类中的某个成员函数通过friend关键字能够访问本类中的私有成员变量，另一个类中的其他成员函数不能访问本类中的私有成员变量。

c++析构函数

发表于 2021-02-26 更新于 2021-03-02 分类于 c++

c++中析构函数跟构造函数是相对于应的，构造函数在对象创建时调用，析构函数在对象销毁时调用。析构函数有如下特点：

  1. 析构函数名字与类名相同，函数名前面需要加上~
  2. 析构函数不带任何参数。（所以不能重载，所以析构函数只有有一个）
  3. 析构函数没有返回值

析构函数一般用于释放构造函数中申请的资源，如动态内存、文件句柄、数据库句柄等。
